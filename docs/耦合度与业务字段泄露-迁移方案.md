# Phinix 耦合度与业务字段泄露 — 未实现项评估与迁移方案

> 2026-05-30，基于《正式发布前置修复清单》《架构耦合度与内聚度评估》《设计哲学》三份文档交叉分析。
> 聚焦：Common 程序集拆分、插件平权、Pipeline 语义完成、外部插件装载、Client 侧抽象层领域耦合。

---

## 1. 总览：未实现项与耦合严重度矩阵

|---|---------|-----------|---------|--------|-----------|---------|
| A | Common 程序集拆分 | 5.5 | 编译期耦合 | **高** | 4 类 + 引用调整 | 12-16h |

## 2. 逐项评估

### 2.1 Common 程序集拆分（Phase 5.5）

**现状：** 4 个端侧实现类编译在 Common 程序集中，通过 multi-target (net472/net10.0) 在编译层面隔离：

| 文件 | 所在程序集 | 实际归属 | 端 |
|------|-----------|----------|-----|
| `Common/Authentication/ClientAuthenticator.cs` | Authentication.dll | Client 侧 | Client |
| `Common/UserManagement/ClientUserManager.cs` | UserManagement.dll | Client 侧 | Client |
| `Common/Connections/NetClient.cs` | Connections.dll | Client 侧 | Client |
| `Common/Connections/NetServer.cs` | Connections.dll | Server 侧 | Server |

**严重度评估：**

当前 multi-target 方案在运行时不会互相污染——net472 产出和 net10.0 产出各自只有对应端的代码路径。但违反设计哲学 4.4：

> 如果一段代码移到某一端，另一端完全不受影响，那它就不该在 Common 里。

具体风险：
- **ClientAuthenticator** 使用 `System.Security.Cryptography`、文件系统 I/O（读写 CredentialStore）、`System.Timers`——纯客户端关注点
- **ClientUserManager** 绑定 `NetClient` + `ClientAuthenticator`，发送 `LoginPacket`、处理 `UserSyncPacket`——纯客户端关注点
- **NetClient** 包含 DNS 解析、probe 多端点连接、客户端侧 poll 线程——纯客户端关注点
- **NetServer** 包含 `connectedPeers` 管理、监听启动——纯服务端关注点

**当前为什么能工作：** multi-target + `#if` 条件编译或运行时路径分叉。但这是靠编译技巧掩盖的架构问题——任何新人阅读 Common 代码时，看到 `NetClient` 和 `NetServer` 在同一目录，会自然地认为它们属于同一层。

**迁移难点：**
1. `ClientAuthenticator` 被 `ClientUserManager` 引用，`ClientUserManager` 被客户端 UI 层引用——需要整体搬迁
2. `NetClient` 被 `ClientAuthenticator` 和 `ClientUserManager` 同时依赖——搬迁顺序有要求
3. Common 中 `Authenticator` / `UserManager` / `NetCommon` 抽象基类需要保留，但子类要迁出——需要新建项目并调整引用链
4. DLL 加载顺序（01-12 编号前缀）需要重新计算——`Assemblies/` 下文件编号可能变化

**结论：** 这是一个干净的机械搬迁，不改变行为，但涉及新建 3-4 个项目、调整 10+ 处引用、回归测试。风险可控但工足量大。

### 2.2 Server 侧插件平权（Phase 5）

**现状：** 6 处违规，Server 在编译期、配置层、启动流程三个层面"认识"Chat 和 Trade：

#### 违规 1：Server.csproj ProjectReference（编译期）

```xml
<!-- Server/Server.csproj:30-32 -->
<ProjectReference Include="..\Extensions\Chat\Contracts\ChatExtension.csproj" />
<ProjectReference Include="..\Extensions\Trade\Contracts\TradeExtension.csproj" />
```

**根因：** Server 需要引用 Chat/Trade 的 Contracts 程序集来使用其中的类型（如配置类、协议常量）。但设计哲学 1.2 明确要求宿主不依赖插件。

**为什么这是真问题：** 如果 Chat 的 Contracts 项目重构或删除，Server 编译失败。新增插件如果也有"Contracts"需要被 Server 消费，就要加第三条 ProjectReference。这是开放-封闭原则的典型违反。

**解决方向：** Server 需要的不是 Chat/Trade 的具体类型，而是"扩展提供的配置"这一抽象能力。应该在 Common 中定义 `IExtensionConfigProvider` 或 Key-Value 配置传递机制，让插件通过注册体系上报配置，而非 Server 直接引用插件类型。

#### 违规 2：MSBuild 构建目标（编译期）

```xml
<!-- Server/Server.csproj:40-62 -->
<OfficialServerExtensionProject Include="..\Extensions\Chat\Server\ChatExtension.Server.csproj" />
<OfficialServerExtensionProject Include="..\Extensions\Trade\Server\TradeExtension.Server.csproj" />
<Target Name="CopyOfficialServerExtensions" AfterTargets="Build">
```

**根因：** Server 构建流程显式编译并复制 Chat/Trade 扩展 DLL。新增官方扩展需要编辑 Server.csproj。

**解决方向：** 将扩展构建与部署完全移出 Server.csproj，交由独立脚本（`scripts/build-extensions.sh` / `scripts/build-extensions.ps1`）负责。Server 的 MSBuild 构建只管自身。详见阶段三 Step 3.4。

#### 违规 3：Config.cs 业务字段（配置耦合）

```csharp
// Server/Config.cs:81,87,117
public string ChatHistoryPath = "chatHistory";
public string TradeDatabasePath = "trades";
public int ChatHistoryLength = 40;
```

**根因：** 这是最典型的"业务字段泄露"——Server 的核心配置类承载了 Chat/Trade 的持久化路径和参数。设计哲学 1.3 要求宿主只做通用服务，不关心具体业务。

**为什么这是真问题：**
- 新增"邮件"插件需要 `MailDatabasePath` → 改 Server Config
- 新增"公告"插件需要 `AnnouncementHistoryLength` → 改 Server Config
- Config 类随插件数量线性膨胀
- 插件的配置默认值和校验逻辑侵入 Server 代码

**解决方向：** 引入 `IExtensionConfigProvider` 接口，让每个扩展在 `Register()` 阶段声明自己的配置项（含默认值、校验规则）。Server Config 只保留通用配置（网络、认证），扩展配置由扩展自行管理，通过 Key-Value store 持久化。

#### 违规 4：Server.cs 启动逻辑（运行时耦合）

```csharp
// Server/Server.cs:62
extensionHostContext.SetOption("builtin.chat.history-capacity", Config.ChatHistoryLength.ToString());
```

**根因：** Server 启动时直接操作 Chat 的内部选项。Server 知道 Chat 的选项键名 `"builtin.chat.history-capacity"` 和对应的配置字段。

**解决方向：** 随违规 3 一起解决。扩展配置由扩展自己在 `Activate()` 中从 Config Provider 读取，而非宿主主动推送。

#### 违规 5：VersionCommand.cs 硬编码（运行时耦合）

```csharp
// Server/Commands/VersionCommand.cs:26-27
Console.WriteLine("FrameworkChat: " + GetAssemblyVersion("ChatExtension.Server"));
Console.WriteLine("FrameworkTrade: " + GetAssemblyVersion("TradeExtension.Server"));
```

**根因：** 版本命令硬编码了特定扩展的程序集名称。新增扩展在版本输出中不可见。

**解决方向：** 改为遍历 `PhinixExtensionRegistry` 中已加载的所有扩展，输出每个扩展的 ID + 版本。`IPhinixExtensionModule` 已有 `ExtensionId`，可新增 `Version` 属性。或者直接遍历 `AppDomain.CurrentDomain.GetAssemblies()` 中所有 Phinix 扩展程序集。

#### 违规 6：Dockerfile 硬编码（部署耦合）

```dockerfile
# Dockerfile:16-18
RUN dotnet restore Server/Server.csproj && \
    dotnet restore Extensions/Chat/Server/ChatExtension.Server.csproj && \
    dotnet restore Extensions/Trade/Server/TradeExtension.Server.csproj
```

**根因：** Docker 构建显式 restore 每个扩展项目。

**解决方向：** 使用通配或脚本化发现——`for each .csproj in Extensions/*/Server/; do dotnet restore $proj; done`。

### 2.3 Client 侧抽象层去领域化（Phase 5）

**现状：** `ClientExtensionAbstractions` 项目存在两类领域耦合：

#### 违规 7：IClientSettingsContext 业务属性（接口耦合）

```csharp
// ClientExtensionAbstractions/Framework/IClientExtensionAbstractions.cs:66-80
int ChatMessageLimit { get; }        // Chat 专用
bool ShowChatFormatting { get; }     // Chat 专用
bool ShowNameFormatting { get; }     // Chat 专用
bool AllItemsTradable { get; }       // Trade 专用
bool ShowBlockedTrades { get; }      // Trade 专用
void BlockUser(string uuid);         // 社交域（Chat 使用）
void UnBlockUser(string uuid);       // 社交域（Chat 使用）
```

**根因：** 设计哲学 1.3 说 host 不关心当前有哪些业务，设计哲学 2.3 说"凡是可能随业务变化的，通过合约抽象"。但 `IClientSettingsContext` 接口直接包含了 Chat 和 Trade 的专用属性。

**为什么这是真问题：**
- 新增"邮件"插件需要 `MailNotificationEnabled` → 改 `IClientSettingsContext`
- 新增"公告"插件需要 `AnnouncementFrequency` → 改 `IClientSettingsContext`
- 接口随插件数量线性膨胀，所有实现者（包括不关心 Chat 的插件）都要实现这些属性
- 任何扩展引用 `IClientSettingsContext` 都会间接依赖 Chat/Trade 领域概念

**解决方向：**
- **方案 A（推荐）：** 将 `IClientSettingsContext` 改为 Key-Value 通用配置接口：
  ```csharp
  T Get<T>(string key, T defaultValue = default);
  void Set<T>(string key, T value);
  ```
  每个扩展使用自己的 key 前缀（如 `"chat.messageLimit"`）读写配置。host 不再定义任何业务字段。

- **方案 B：** 将 Chat/Trade 专用属性拆到各自扩展的 Contracts 中，作为独立接口（如 `IChatSettings`、`ITradeSettings`），扩展通过 API Registry 互相解析。但这增加了跨扩展依赖。

#### 违规 8：IChatTabContent 位置错误（命名空间污染）

```csharp
// ClientExtensionAbstractions/UI/IChatTabContent.cs
public interface IChatTabContent
{
    void Draw(Rect inRect);
    void ScrollToBottom();
}
```

**根因：** 这是 Chat 专用接口，却放在通用抽象层中。设计哲学 1.1 要求插件平权——不应有一个插件的接口被提升到"框架级别"。

**解决方向：** 迁至 `Extensions/Chat/Contracts/`。检查所有引用方——目前只有 Chat 自身使用此接口（`ChatMainTabProvider` 返回它）。如果 `ServerTab` 容器通过此接口与 Chat Tab 交互，应先抽象为通用接口（如 `ITabContent { void Draw(Rect); }`），让 `IChatTabContent` 继承它。

#### 违规 9：ClientExtensionAbstractions.csproj 绑定游戏引擎

```xml
<!-- ClientExtensionAbstractions.csproj -->
<Reference Include="Assembly-CSharp" />
<Reference Include="UnityEngine.CoreModule" />
<Reference Include="UnityEngine.IMGUIModule" />
<Reference Include="UnityEngine.TextRenderingModule" />
<Reference Include="UnityEngine" />
```

**根因：** "抽象层"本应与运行时无关，但为了在接口签名中使用 `Rect`、`Window` 等类型，直接引用了 UnityEngine 和 Assembly-CSharp（RimWorld）。

**评估：** 这是一个务实妥协——如果接口签名需要 `Rect`（如 `Draw(Rect inRect)`），就不可能完全脱离 UnityEngine。但应该：
1. 在项目命名/文档中明确标注"客户端专用抽象层"
2. 尽可能减少引擎类型在接口签名中的出现（如用自定义 `struct PmiRect` 替代 `UnityEngine.Rect`）
3. 长期目标：将真正不需要引擎的接口（如 `IClientSettingsContext`、`IClientSessionContext`）抽到 `Common` 中

#### 违规 10：注册模式不一致（低）

Chat 的 `IMainTabProvider` 在 `Activate()` 中注册，Trade 的在 `Register()` 中注册。两种方式都能工作但含义不同。按设计意图，`Register()` 声明"我提供什么 API"，`Activate()` 做需要 host context 的初始化。Chat 应统一到 `Register()`。

### 2.4 Client Settings 业务字段迁移（Phase 5）

**现状：** [Client/Source/Settings.cs](../Client/Source/Settings.cs) 是一个 300 行的单体类，混合了：

| 类别 | 字段 | 行号 |
|------|------|------|
| 通用 | `ServerAddress`, `ServerPort`, `DisplayName` | 15-29 |
| Chat | `ShowNameFormatting`, `ShowChatFormatting`, `ShowUnreadMessageCount`, `ShowBlockedUnreadMessageCount`, `ChatMessageLimit`, `ForceMessageFieldFocus` | 46-94 |
| Trade | `AcceptingTrades`, `AllItemsTradable`, `ShowBlockedTrades`, `DropCurrentMap` | 39-130 |
| 社交 | `BlockedUsers`, `CollapseBlockedUsers` | 127-138 |
| 内部 | `Migrated` | 118 |

**根因：** 与 Server Config 同样的问题——宿主设置文件承载了所有业务字段。`ExposeData()` 序列化全部 16 个字段为一个平铺列表，`IsChanged` 比较全部。

**解决方向：** 随违规 7 一起解决——将 `IClientSettingsContext` 改为 Key-Value 通用接口后，Settings.cs 只需保留 `ServerAddress`、`ServerPort`、`DisplayName` 三个通用字段。Chat/Trade 设置由各自扩展通过 Key-Value 接口读写，各自管理持久化键名和默认值。

### 2.5 ServerPipeline 五段语义完成（Phase 4）

**现状：** `ServerPipelineRunner` 已实现 Message/Command 的五段管道（IngressValidation → InboundInterception → DefaultProcess → Observation → OutboundInterception），但有三项缺失：

#### 缺失 1：Item 管道未集成

`IItemCodec` 接口已定义，`DiscoveredPhinixExtensions.ItemCodecs` 列表已被收集和排序，但 `ServerPipelineRunner` 中没有 Item 管道的执行逻辑。`PhinixFrameworkServer` 在 Item 分支只做序列化，不经过编解码器。

**影响：** 低——当前无插件使用 Item 管道。但接口已暴露，若不实现会在第一个 Item 插件出现时成为阻塞项。

#### 缺失 2：LegacyFallback 枚举值未处理

`MessageHandlingResultAction.LegacyFallback` 存在但 `ServerPipelineRunner` 中的所有处理逻辑都不检查它。落入此路径的数据包被静默视为 `Continue`。

**影响：** 低——当前无代码返回 `LegacyFallback`。但枚举值存在却不处理，是接口契约不完整的信号。

#### 缺失 3：运行时优先级排序未强制

`ServerPipelineRunner` 暴露公共可写 `IList<T>` 属性。排序完全依赖 `PhinixExtensionRegistry` 在构建时排序。如果运行时向这些列表添加项，不会重新排序。

**影响：** 低——当前没有运行时添加的场景。但封装不完整，未来可能成为 bug。

**结论：** 三项缺失影响均低。Item 管道是唯一需要新代码的项，其余两项是防御性修补。可以一并快速完成。

### 2.6 外部插件目录扫描与程序集装载（Phase 5）

**现状：** 机制已就绪（`ExtensionAssemblyLoader.LoadAssemblies` + `PhinixExtensionRegistry.DiscoverExtensions`），但以下问题使其不能算"完成"：

#### 已实现：
- `Extensions/` 和 `UserExtensions/` 目录扫描
- `Assembly.LoadFrom` + `AssemblyResolve` 依赖解析
- `IPhinixExtensionModule` 自动发现与注册
- 旧版 `[PhinixExtension]` 属性兼容

#### 缺失：
1. **构建期绑定未消除**（与违规 1、2 重叠）——Server 仍通过 ProjectReference 编译期依赖 Chat/Trade Contracts
2. **Dockerfile 硬编码**（违规 6）——构建脚本写死扩展名称
3. **无插件清单**——推迟到 2.0，当前可接受
4. **无动态热加载**——推迟到 2.0，当前可接受
5. **探测目录硬编码**——`GetExtensionProbeDirectories()` 返回固定 `Extensions/` 和 `UserExtensions/`
6. **目录不存在时静默跳过**——不创建目录，不报警

**结论：** 运行时机制已完成，但编译期和部署期绑定使"外部插件"名不副实——当前 Chat/Trade 仍是编译时硬依赖，无法真正做到"新增插件不改 Server"。Phase 5 的真正完成标志是违规 1-2-6 的解决。

## 3. 迁移方案

按"每阶段可编译、可运行、可验证"的渐进式原则，分 4 个阶段推进。

### 阶段一：Pipeline 收尾 + 低风险清理（4-6h，可立即开始）

本阶段不改变任何边界，只补全已有接口的缺失实现。风险最低，可独立验证。

#### Step 1.1：Item 管道集成（2h）

**文件：** `Server/ServerRuntime/ServerPipelineRunner.cs`

新增 `ProcessIncomingItem` 方法：
```
Inbound Item → foreach IItemCodec → Decode → DefaultItemHandler → Encode → Outbound
```

在 `PhinixFrameworkServer` 中，将 Item 分支从"直接序列化"改为"通过 PipelineRunner 处理"。

**涉及接口：** `IItemCodec`（已定义）、`IServerDefaultItemHandler`（可能需要新增，或复用 `IServerDefaultCommandHandler`）

#### Step 1.2：LegacyFallback 处理（0.5h）

**文件：** `Server/ServerRuntime/ServerPipelineRunner.cs`

在 `isHandled` / `isBlocked` / `shouldContinue` 辅助方法中添加 `LegacyFallback` 分支，行为等效于 `Continue` 但记录 Warning 日志。

#### Step 1.3：Pipeline 列表封装（1h）

**文件：** `Server/ServerRuntime/ServerPipelineRunner.cs`

将公共 `IList<T>` 属性改为 `IReadOnlyList<T>`，通过构造函数注入并在内部排序。移除运行时的可变性。

#### Step 1.4：Chat IMainTabProvider 注册位置修正（0.5h）

**文件：** `Extensions/Chat/Client/BuiltInChatClientExtension.cs`

将 `builder.RegisterApi<IMainTabProvider>(chatMainTabProvider)` 从 `Activate()` 移至 `Register()`。

---

### 阶段二：Client 侧抽象层去领域化（10-14h，依赖阶段一完成）

核心目标：消除 `ClientExtensionAbstractions` 中的业务字段泄露。这是耦合度改进中影响面最大的部分。

#### Step 2.1：IClientSettingsContext 改为 Key-Value 通用接口（4h）

**文件改动：**

1. **`ClientExtensionAbstractions/Framework/IClientExtensionAbstractions.cs`**
   - 将 `IClientSettingsContext` 接口简化为：
   ```csharp
   public interface IClientSettingsContext
   {
       T Get<T>(string key, T defaultValue = default);
       void Set<T>(string key, T value);
       event Action<string, object> OnSettingChanged;
   }
   ```
   - 移除 `ChatMessageLimit`、`ShowChatFormatting`、`ShowNameFormatting`、`AllItemsTradable`、`ShowBlockedTrades`、`BlockUser`、`UnBlockUser`

2. **`Client/Source/Settings.cs`**
   - 改造 `ExposeData()`：通用字段（`ServerAddress`、`ServerPort`、`DisplayName`）照常序列化；业务字段改为序列化一个 `Dictionary<string, object> ExtensionSettings`
   - 保留 `BlockedUsers` 的持久化（作为通用社交功能保留在宿主，或迁移到 Chat 插件）
   - 实现新的 `IClientSettingsContext` 接口，`Get<T>` / `Set<T>` 从 `ExtensionSettings` 字典读写

3. **所有消费者适配（3 个文件）：**
   - `Extensions/Chat/Client/ChatUiHostContext.cs` — `ChatMessageLimit` → `settings.Get<int>("chat.messageLimit", 100)`
   - `Extensions/Chat/Client/ChatMessageList.cs` — `ShowChatFormatting` → `settings.Get<bool>("chat.showFormatting", true)`
   - `Extensions/Trade/Client/TradeWindow.cs` — `AllItemsTradable` → `settings.Get<bool>("trade.allItemsTradable", false)`
   - `Extensions/Trade/Client/BuiltInTradeClientExtension.cs` — `ShowBlockedTrades` → `settings.Get<bool>("trade.showBlocked", true)`

#### Step 2.2：IChatTabContent 迁出通用抽象层（2h）

1. **`Extensions/Chat/Contracts/`** — 新建 `IChatTabContent.cs`
2. **`ClientExtensionAbstractions/UI/IChatTabContent.cs`** — 替换为通用 `ITabContent { void Draw(Rect inRect); }`（如果要保留通用 Draw 能力）
3. **检查 `ServerTab` 容器**是否通过 `IChatTabContent` 与 Chat Tab 交互——如果是，改为通过 `ITabContent` 或直接通过 `IMainTabProvider`

#### Step 2.3：ClientExtensionAbstractions 命名与文档澄清（1h）

#### Step 3.4：扩展构建与部署改用外部脚本（2h）

**设计决策：** 采用方案 B——将扩展构建和拷贝完全移出 Server.csproj，交由独立的 bash 和 pwsh 脚本负责。Server 的 MSBuild 构建只管自身，不再知道任何扩展的存在。

**选取方案 B 而非方案 A（MSBuild 通配符）的理由：**

| 维度 | 方案 A（通配符） | 方案 B（外部脚本） |
|------|-----------------|-------------------|
| 解耦程度 | 部分解耦——Server 构建仍负责扩展编译 | **完全解耦**——Server 构建与扩展构建是两个独立步骤 |
| 新增扩展 | 自动发现，但通配符范围不可控 | 显式调用脚本，或脚本内自动遍历 |
| 调试体验 | 异常混在 MSBuild 输出中，难以定位 | 独立运行，输出干净 |
| CI 集成 | 隐藏在 MSBuild 生命周期里 | CI 可独立调用同一脚本 |
| 扩展单独构建 | 不支持 | `--extension Chat` 只构建指定扩展 |

**脚本职责：**

- 扫描 `Extensions/*/Server/*.csproj`，自动发现所有扩展项目
- 按依赖顺序构建（Chat → Trade，后续引入依赖图时在此维护）
- 复制输出 DLL 到 `Server/bin/<Configuration>/net10.0/Extensions/`
- 支持 `--extension <name>` 单独构建某个扩展
- 支持 `--clean` 清理扩展输出

**新增文件：`scripts/build-extensions.sh`**

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
REPO_ROOT="$(dirname "$SCRIPT_DIR")"
CONFIGURATION="${CONFIGURATION:-Release}"
TARGET_FRAMEWORK="net10.0"
EXTENSIONS_DIR="$REPO_ROOT/Extensions"
OUT_DIR="$REPO_ROOT/Server/bin/$CONFIGURATION/$TARGET_FRAMEWORK/Extensions"

# 扩展构建顺序（依赖低的先构建）
# 后续可改为从 plugin.json 读取依赖图自动排序
EXTENSION_ORDER=(
    "Chat"
    "Trade"
)

build_extension() {
    local name="$1"
    local proj="$EXTENSIONS_DIR/$name/Server/${name}Extension.Server.csproj"

    if [[ ! -f "$proj" ]]; then
        echo "[ERROR] Extension '$name' project not found: $proj"
        return 1
    fi

    echo "[BUILD] $name..."
    dotnet build "$proj" \
        -c "$CONFIGURATION" \
        --no-restore \
        /p:TargetFramework="$TARGET_FRAMEWORK"

    local dll_src="$EXTENSIONS_DIR/$name/Server/bin/$CONFIGURATION/$TARGET_FRAMEWORK/${name}Extension.Server.dll"
    mkdir -p "$OUT_DIR"
    cp "$dll_src" "$OUT_DIR/"
    echo "[OK] $name → $OUT_DIR/"
}

clean_extensions() {
    echo "[CLEAN] Removing extension outputs..."
    rm -rf "$OUT_DIR"
    mkdir -p "$OUT_DIR"
    echo "[OK] Cleaned."
}

# --- main ---
cd "$REPO_ROOT"

if [[ "${1:-}" == "--clean" ]]; then
    clean_extensions
    exit 0
fi

# 先 restore 所有扩展
for name in "${EXTENSION_ORDER[@]}"; do
    proj="$EXTENSIONS_DIR/$name/Server/${name}Extension.Server.csproj"
    dotnet restore "$proj" > /dev/null
done

if [[ -n "${1:-}" ]] && [[ "$1" != "--all" ]]; then
    # 单扩展模式
    build_extension "$1"
else
    # 全量模式
    for name in "${EXTENSION_ORDER[@]}"; do
        build_extension "$name"
    done
fi

echo ""
echo "=== All extensions deployed ==="
```

**新增文件：`scripts/build-extensions.ps1`**

```powershell
param(
    [string]$Extension = "",
    [switch]$Clean = $false,
    [string]$Configuration = "Release"
)

$ErrorActionPreference = "Stop"
$ScriptDir = Split-Path -Parent $MyInvocation.MyCommand.Path
$RepoRoot = Split-Path -Parent $ScriptDir
$TargetFramework = "net10.0"
$ExtensionsDir = Join-Path $RepoRoot "Extensions"
$OutDir = Join-Path $RepoRoot "Server" "bin" $Configuration $TargetFramework "Extensions"

# 扩展构建顺序（依赖低的先构建）
$ExtensionOrder = @("Chat", "Trade")

function Build-Extension($Name) {
    $Proj = Join-Path $ExtensionsDir $Name "Server" "${Name}Extension.Server.csproj"

    if (-not (Test-Path $Proj)) {
        Write-Error "Extension '$Name' project not found: $Proj"
        return
    }

    Write-Host "[BUILD] $Name..." -ForegroundColor Cyan
    dotnet build $Proj `
        -c $Configuration `
        --no-restore `
        /p:TargetFramework="$TargetFramework"
    if ($LASTEXITCODE -ne 0) { throw "Build failed for $Name" }

    $DllSrc = Join-Path $ExtensionsDir $Name "Server" "bin" $Configuration $TargetFramework "${Name}Extension.Server.dll"
    New-Item -ItemType Directory -Force -Path $OutDir | Out-Null
    Copy-Item $DllSrc $OutDir -Force
    Write-Host "[OK] $Name → $OutDir/" -ForegroundColor Green
}

# --- main ---
Set-Location $RepoRoot

if ($Clean) {
    Write-Host "[CLEAN] Removing extension outputs..." -ForegroundColor Yellow
    Remove-Item -Recurse -Force $OutDir -ErrorAction SilentlyContinue
    New-Item -ItemType Directory -Force -Path $OutDir | Out-Null
    Write-Host "[OK] Cleaned." -ForegroundColor Green
    return
}

# 先 restore 所有扩展
foreach ($Name in $ExtensionOrder) {
    $Proj = Join-Path $ExtensionsDir $Name "Server" "${Name}Extension.Server.csproj"
    dotnet restore $Proj | Out-Null
}

if ($Extension) {
    Build-Extension $Extension
} else {
    foreach ($Name in $ExtensionOrder) {
        Build-Extension $Name
    }
}

Write-Host ""
Write-Host "=== All extensions deployed ===" -ForegroundColor Green
```

## 6. 风险与注意事项

1. **Settings/S Config 兼容性**：改为 Key-Value 后，已有用户的配置文件（XML 序列化）需要迁移逻辑，否则设置丢失。给 `Migrated` 字段分配迁移任务。

2. **DLL 加载顺序变更**：阶段四调整编号前缀后，RimWorld `ModAssemblyHandler` 可能因加载顺序变化抛出 `ReflectionTypeLoadException`。必须逐序验证。

3. **Server ProjectReference 移除**：如果 Server 代码中有未发现的隐式类型引用（如 lambda 中捕获了 `ChatExtension` 类型），移除引用后编译失败。排查需彻底。

4. **ClientExtensionAbstractions 接口变更**：`IClientSettingsContext` 改变签名后，所有实现者和消费者都要适配。影响面包括 Chat、Trade、RedPacket 和宿主自身。用编译错误驱动适配，不会遗漏。

5. **渐进式提交**：每阶段单独一个 commit，附清晰的 commit message 说明"改了什么、为什么、如何验证"。不做跨阶段大提交。
