# Phinix 架构耦合度与内聚度评估

> 对照 `cs架构重构和技术迁移需求分析-基于当前实现修订版.md` 和 `设计哲学.md`，对当前代码库的耦合度与内聚度进行评估。
> 2026-05-29（修订版：新增 UI 性能、内存泄漏、网络错误处理专项审查）

---

## 1. 总体判断

当前处于 **Phase 2.5 ~ Phase 4 过渡态**，架构质量明显好于文档描述的"最坏情况"。文档中描述的六类核心问题（Common 过厚、Server Mono、Common 编译 Server 源码、客户端 Unity 依赖污染 Common、host 强依赖官方扩展 API、无统一 pipeline）中，**四项已得到实质性解决或大幅改善**。

剩余问题集中在"共享层的端侧实现仍未完全拆分"和"插件平权的最后一公里"。

## 2. 高内聚（良好）方面

### 2.1 层次化依赖方向已基本正确

```
插件层 (Extensions/Chat, Extensions/Trade)
  → 共享契约层 (ClientExtensionAbstractions, Common/Utils/Framework)
    → 基础设施层 (Common/Utils, Common/Connections, Common/Authentication, Common/UserManagement)
```

- [Client.csproj](../Client/Source/Client.csproj) **零引用** ChatExtension 或 TradeExtension 项目 —— 完全符合设计哲学 1.2（host 不依赖插件）
- 宿主通过 `ResolveExtensionApis<IMainTabProvider>()` / `ResolveExtensionApis<IServerSidebarProvider>()` 动态收集插件能力，不做类型强绑定
- Common csproj **不再有** `Compile Include ../../Server/*.cs` —— 编译方向已拉直，引用方向为 `Server → Common`，而非 `Common → 编译 Server 源码`
- Common 项目已 multi-target：`net472;net10.0`，Client 消费 net472 输出，Server 消费 net10.0 输出

### 2.2 插件化注册体系已统一

Chat 和 Trade 均通过 `IPhinixExtensionModule` 的 `Register(builder)` / `Activate(hostContext)` 接入系统：

- 同一套发现路径（`PhinixExtensionRegistry.DiscoverExtensions`）
- 同一套注册路径（`builder.RegisterApi<T>()` / `builder.AddClientMessageHandler()`）
- 同一套 API registry（`RegisterApi<T>()` / `ResolveAll<T>()`）
- 同一套 UI 挂载点（`IMainTabProvider` / `IServerSidebarProvider` / `IBadgeProvider`）

三类处理链入口已落地：message / command / item handler。

### 2.3 Phase 4 Pipeline 接口已定义并实现

[FrameworkTypes.cs](../Common/Utils/Framework/FrameworkTypes.cs) 中已包含完整接口层级：

| 接口 | 职责 | 状态 |
|------|------|------|
| `IServerInboundMessageInterceptor` | 入站消息拦截 | 已定义 |
| `IServerDefaultMessageHandler` | 默认消息处理 | 已定义 |
| `IServerMessageObserver` | 后置消息观察 | 已定义 |
| `IServerInboundCommandInterceptor` | 入站命令拦截 | 已定义 |
| `IServerDefaultCommandHandler` | 默认命令处理 | 已定义 |
| `IServerCommandObserver` | 后置命令观察 | 已定义 |
| `IServerOutboundPacketInterceptor` | 出站拦截 | 已定义 |

`ServerPipelineRunner` 已实现 IngressValidation → InboundInterception → DefaultProcess → Observation → OutboundInterception 完整执行链。

### 2.4 Server 现代化迁移基本完成

- Server + 所有 Server 侧子项目（Authentication.Server、UserManagement.Server、ServerRuntime）均已迁移到 **net10.0**
- Dockerfile 使用 `mcr.microsoft.com/dotnet/sdk:10.0` 构建 + `mcr.microsoft.com/dotnet/runtime:10.0` 运行
- 不再依赖 mono，使用原生 `dotnet` 命令
- Common 项目 multi-target `net472;net10.0`，两端独立消费

### 2.5 ClickOnce 构建链 DLL 排序

Client 侧后构建步骤已按依赖顺序分配编号前缀（01-LiteNetLib ~ 12-PhinixClient），字符串序 = 加载序，RimWorld `ModAssemblyHandler` 不会抛出 `ReflectionTypeLoadException`。

## 3. 耦合问题

### 3.1 Common 程序集仍承载端侧实现（中等）

虽然物理目录已整理（Extensions/ 独立、不再反向编译 Server 源码），但以下 client-only / server-only 实现仍然编译在 Common 程序集中：

| 文件 | 所在程序集 | 实际归属 |
|------|-----------|----------|
| [ClientAuthenticator.cs](../Common/Authentication/ClientAuthenticator.cs) | Authentication.dll | 应拆至 Client 侧或独立 Authentication.Client |
| [ClientUserManager.cs](../Common/UserManagement/ClientUserManager.cs) | UserManagement.dll | 应拆至 Client 侧或独立 UserManagement.Client |
| [NetClient.cs](../Common/Connections/NetClient.cs) | Connections.dll | 应拆至 Client 侧或独立 Connections.Client |
| [NetServer.cs](../Common/Connections/NetServer.cs) | Connections.dll | 应拆至 Server 侧或独立 Connections.Server |

这些类型当前通过 multi-target (net472 / net10.0) 在编译层面做了隔离，运行时不会互相污染——每个目标框架只产出该端需要的代码路径。但从"Common 只保留双端共享实现"的标准看，它们仍属于应拆项，对应文档 Phase 5.5。

### 3.2 ClientExtensionAbstractions 存在领域耦合（中等）

在 [IClientExtensionAbstractions.cs](../Client/ClientExtensionAbstractions/Framework/IClientExtensionAbstractions.cs) 中：

```csharp
// 第 66-74 行 — chat/trade 业务语义侵入通用设置接口
int ChatMessageLimit { get; }
bool ShowChatFormatting { get; }
bool AllItemsTradable { get; }
bool ShowBlockedTrades { get; }
```

这是"通用抽象层知道具体业务领域"的典型耦合信号。按设计哲学 4.3，host 不应为特定插件保留专用设置项。这些业务设置应由各自插件自行管理，或通过泛型 Key-Value 机制传递。

同时 [IChatTabContent.cs](../Client/ClientExtensionAbstractions/UI/IChatTabContent.cs) 是一个 chat 专用接口，却放在了通用抽象层。应迁至 `Extensions/Chat/Contracts`。

另外 `ClientExtensionAbstractions.csproj` 直接依赖 `Assembly-CSharp`、`UnityEngine.*`，源文件中含 `using Verse;`——这使得"抽象层"实际上绑定了客户端运行时。如果只是编译依赖（接口签名需要引用 `Rect`、`Window` 等 UnityEngine/Verse 类型），短期可以接受，但应明确其定位是**客户端专用抽象层**而非双端共享抽象层，命名上应有所体现。

### 3.3 注册模式不一致（低）

- `ChatMainTabProvider` 在 `Activate()` 中通过 `hostContext.ApiRegistry.RegisterApi<IMainTabProvider>(this)` 注册
- `TradeMainTabProvider` 在 `Register()` 中通过 `builder.RegisterApi<IMainTabProvider>(this)` 注册

两种方式都能工作。按设计意图，`Register()` 更适合声明"我提供什么 API"，`Activate()` 更适合做需要 host context 的初始化。应将 Chat 的 `IMainTabProvider` 注册也移至 `Register()`。

### 3.4 Server 构建时显式编译官方扩展（低）

```xml
<!-- Server/Server.csproj 第40-62行 -->
<OfficialServerExtensionProject Include="..\Extensions\Chat\Server\ChatExtension.Server.csproj" />
<OfficialServerExtensionProject Include="..\Extensions\Trade\Server\TradeExtension.Server.csproj" />
<Target Name="CopyOfficialServerExtensions" AfterTargets="Build">
```

构建脚本在 Server 构建后显式编译并复制 Chat/Trade Server 扩展 DLL。这是 Phase 5 文档中定义的"过渡措施"——构建时复制 DLL 到宿主输出目录。Phase 5 的目标是改为目录扫描 + 运行时装载。当前阶段可接受。

### 3.5 Client 侧 Settings 仍有业务字段（低）

[Client/Source/Settings.cs](../Client/Source/Settings.cs) 中包含 `AcceptingTrades`、`ShowChatFormatting`、`ChatMessageLimit`、`ShowNameFormatting`、`ForceMessageFieldFocus`、`AllItemsTradable`、`ShowBlockedTrades` 等业务相关字段。这些字段的存在意味着宿主仍然"认识"聊天和交易这两个具体业务领域，离"宿主完全不知道当前有哪些业务"还有一步。

### 3.6 Server 宿主仍持有 Chat/Trade 的业务知识（中）

以下违规均属于"宿主知道具体业务扩展"的模式：

| # | 文件 | 问题 |
|---|------|------|
| 1 | [Server/Server.csproj:30,32](../Server/Server.csproj) | ProjectReference 引用 ChatExtension、TradeExtension Contracts |
| 2 | [Server/Server.csproj:40-62](../Server/Server.csproj) | 专用 MSBuild 目标和项目组为 Chat/Trade 构建和复制 DLL |
| 3 | [Server/Config.cs:78-87,114-117](../Server/Config.cs) | `ChatHistoryPath`、`TradeDatabasePath`、`ChatHistoryLength` 等业务配置字段 |
| 4 | [Server/Server.cs:62](../Server/Server.cs) | 宿主启动时为 Chat 设置 `builtin.chat.history-capacity` 选项 |
| 5 | [Server/Commands/VersionCommand.cs:26-27](../Server/Commands/VersionCommand.cs) | 硬编码 `ChatExtension.Server`、`TradeExtension.Server` 程序集名称 |
| 6 | [Dockerfile:17-18](../Dockerfile) | 硬编码 Chat/Trade 扩展的 `dotnet restore` 步骤 |

这些在"构建时可用的过渡态"下可接受，但应在 Phase 5 中逐步消除。

对照 RimWorld 每帧调用 `DoWindowContents` / `DoButton` / `Draw` 的触发模型，以下问题增加了 GC 压力并降低渲染帧率：

### 4.2 中等问题

| 严重度 | 文件:行号 | 问题 |
|--------|-----------|------|
| MEDIUM | [SettingsWindow.cs:79-83](../Client/Source/GUI/Windows%20and%20panels/SettingsWindow.cs) | 每帧 foreach + CalcHeight 遍历所有设置项 —— 无变化时也应跳过 |

### 4.3 根因分析

UI 渲染问题根源是 **布局计算与渲染未分离**。`CalcHeight`、`CalcWidth` 和 LINQ 聚合本应在数据变更时执行一次并缓存，但目前被嵌入 `Draw` 路径每帧重新计算。Chat/Trade UI 扩展直接将业务数据查询混入渲染循环（如 `hostContext.ChatService.UnreadMessages` 在 `BadgeText` getter 中每帧调用），进一步模糊了数据层与呈现层的边界。

## 5. 内存与生命周期管理（新增）

### 5.1 严重 Bug：Session 字典键不匹配导致泄漏

**文件：** [Server/Authentication/ServerAuthenticator.cs:336](../Server/Authentication/ServerAuthenticator.cs)

```csharp
// BUG: sessions 字典的键是 connectionId，但 Remove 使用了 session.SessionId
sessions.Remove(session.SessionId);  // 应改为 sessions.Remove(e.ConnectionId)
```

导致断开连接的 session 条目无法从字典中移除，成为孤儿条目。只有在 30 分钟超时后才会被清理定时器回收。

### 5.2 Timer 泄漏

| 文件 | 变量 | 问题 |
|------|------|------|
| [Server/Authentication/ServerAuthenticator.cs:74-80](../Server/Authentication/ServerAuthenticator.cs) | `sessionCleanupTimer` | 永不停止/释放，运行到进程结束 |
| [Common/Authentication/ClientAuthenticator.cs:119-131](../Common/Authentication/ClientAuthenticator.cs) | `sessionExtendTimer` | 停止了但从未 Dispose |
| [Server/Server.cs:80-87](../Server/Server.cs) | `saveTimer` (static) | 停止了但从未 Dispose |
| [Client/Source/Framework/PhinixFrameworkClient.cs:56-62](../Client/Source/Framework/PhinixFrameworkClient.cs) | `negotiationTimer` | 停止了但从未 Dispose |

### 5.3 未实现 IDisposable 的资源持有类

以下类持有 `Timer`、`NetManager`、`Thread` 等可释放资源，但未实现 `IDisposable`：

- [NetClient.cs](../Common/Connections/NetClient.cs) — `NetManager`、poll 线程、probe 线程
- [NetServer.cs](../Common/Connections/NetServer.cs) — `NetManager`、poll 线程
- [ServerAuthenticator.cs](../Server/Authentication/ServerAuthenticator.cs) — `Timer`
- [ClientAuthenticator.cs](../Common/Authentication/ClientAuthenticator.cs) — `Timer`
- [PhinixFrameworkClient.cs](../Client/Source/Framework/PhinixFrameworkClient.cs) — `Timer`

### 5.4 客户端扩展 Shutdown 从未调用

`PhinixFrameworkClient` 从未调用 `PhinixExtensionRegistry.ShutdownExtensions()`。这意味着 `BuiltInChatClientExtension.Shutdown()`、`BuiltInTradeClientExtension.Shutdown()` 和 `PhinixDefaultTradeBehaviour.Stop()` 中的 `-=` 取消订阅代码从未执行。对于 RimWorld 的进程生命周期模型来说影响较小（进程退出时 OS 回收），但 Cleanup 代码实际上是死代码。

### 5.5 事件订阅未解除

以下组件订阅了网络事件但从未解除（如果宿主对象被替换将导致泄漏）：

- `ServerAuthenticator` → `netServer.OnConnectionEstablished/OnConnectionClosed/RegisterPacketHandler`
- `ServerUserManager` → `netServer.OnConnectionClosed/RegisterPacketHandler`
- `PhinixFrameworkServer` → `netServer.OnConnectionClosed/RegisterPacketHandler`
- `ClientAuthenticator` → `netClient.OnDisconnect/RegisterPacketHandler`
- `ClientUserManager` → `netClient.OnDisconnect/RegisterPacketHandler`
- `PhinixFrameworkClient` → `netClient.OnDisconnect/RegisterPacketHandler`

在单例模式下不会触发实际泄漏，但增加了重构和测试的耦合风险。

## 8. 三类专项审查的综合诊断

### 8.1 UI 性能：计算与呈现耦合

根因是 **布局计算（CalcHeight/CalcWidth）被嵌入渲染循环（Draw）**。在数据驱动的架构中，布局应在数据变更时计算一次并缓存，当前每帧重新计算导致无谓的 LINQ 分配、Regex 编译和 Text.CalcHeight 调用。修复方向：
- 引入脏标记（dirty flag），仅在数据变更时重新计算布局
- 将 `TextHelper.StripRichText` 的 Regex 预编译为 `static readonly`
- 将 `BadgeText` 等属性 getter 中的实时查询改为推送式更新
- 消除 `new TextWidget(...)` 等每帧分配

### 8.2 内存泄漏：生命周期管理缺位

核心模式是 **资源持有者未声明所有权**。Timer、NetManager、Thread 等 IDisposable 资源被字段持有但封闭类未实现 IDisposable。事件订阅是单向的（只 += 不 -=），依赖进程退出作为终极清理手段。修复方向：
- 实现 IDisposable 模式（至少对于持有 Timer/NetManager 的类）
- 配对事件订阅与取消订阅（或在 `Shutdown` 中统一清理）
- 修复 `ServerAuthenticator.cs:336` 的键不匹配 Bug

### 8.3 网络错误处理：静默降级链

整个网络栈中，错误倾向于被静默消费而非上报。异常被捕获并记录日志后即丢弃，调用方无感知。后台线程直接修改共享状态，部分关键数据结构（`connectedPeers`、`probePeers`）无同步保护。修复方向：
- 为关键握手包（Hello/Auth/Login/SessionExtend）添加重试或超时逻辑
- 为 `connectedPeers` 和 `probePeers` 添加锁保护
- 将网络回调中的事件触发封送至主线程
- 引入反压机制（限制 `displayMessages` 和 `pendingActions` 大小）
