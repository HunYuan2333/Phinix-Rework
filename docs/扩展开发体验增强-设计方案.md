# Phinix 扩展开发体验增强 — 设计方案

> 2026-05-30，基于解耦评估成果。目标：让外部开发者在零 Phinix 源码改动的前提下，能开发、调试、发布独立创意工坊 Mod。

---

## 1. 背景与当前状态

Phinix 的解耦目标已达成——外部 Mod 通过 `IPhinixExtensionModule` + `[PhinixExtension]` 即可完整接入，无需修改 Phinix 源码。但在开发体验层面存在以下缺口：

### 1.1 已就绪的能力（无需改动）

| 能力 | 说明 |
|------|------|
| 自动发现 | `PhinixExtensionRegistry.DiscoverExtensions()` 遍历 `AppDomain.CurrentDomain.GetAssemblies()`，通过 `GetReferencedAssemblies()` 判断是否引用了 `Utils.dll`，从而自动发现外部 Mod 的扩展模块 |
| 生命周期 | `Register()` → `Activate()` → `Shutdown()` 三阶段，通过 `IExtensionBuilder` / `ExtensionHostContext` 接入框架服务 |
| UI 挂载点 | `IMainTabProvider`（Tab）、`IServerSidebarProvider`（侧栏）、`IBadgeProvider`（角标）——三者均可被外部 Mod 注册 |
| 通信接入 | `IClientMessageHandler`、`IClientCommandHandler`、`IMessageRenderer`、`ICapabilityProvider` |
| 框架服务 | `IClientSettingsContext`（Key-Value 持久化设置）、`IClientSessionContext`、`IClientUserDirectory`、`IClientUserEventStream`、`IClientMainThreadDispatcher`、`IClientWindowService`、`IClientSoundService` |
| 跨插件交互 | `IExtensionApiRegistry` — 插件通过 `builder.RegisterApi<T>()` 暴露能力，其他插件通过 `builder.TryResolveApi<T>()` 消费 |
| 数据持久化 | `builder.HostContext.RegisterPersistent()` — 服务端扩展数据自动随 `SaveInterval` 保存/加载 |
| 服务端配置 | `IExtensionConfigProvider` + `IExtensionConfigSection` — 扩展配置 Key-Value 持久化到 `server.conf` 的 `ExtensionConfigs` |

### 1.2 当前缺口

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 扩展加载失败静默 | **高** | `Diagnostics` 走 `LogLevel.DEBUG`，非 DevMode 不可见。开发者不知道自己的扩展为什么没加载 |
| 无扩展管理面板 | **高** | 玩家/开发者无法查看哪些扩展已加载、哪些失败、状态如何 |
| 无扩展配置入口 | **中** | 扩展设置散落在 `SettingsWindow`（仅 Phinix 官方扩展）或各自实现 Tab，没有统一入口 |
| 无版本兼容性检查 | **低** | 框架 API 未来 breaking change 时，旧扩展会抛 `MissingMethodException` 而非清晰的版本不兼容提示 |
| 无开发模板 | **低** | 新开发者需要自己拼 `.csproj`、引用链、最小可编译结构 |
| 扩展热重载 | **无（推迟到 2.0）** | 当前架构下修改扩展需要重启 RimWorld |

## 2. 扩展管理面板设计

### 2.1 目标

在 Phinix 自身的 `ServerTab` 中新增一个 **"扩展" Tab**（或作为现有 Tab 的子面板），展示：

- 所有已发现扩展的列表（名称、ID、状态、版本）
- 加载诊断信息（警告/错误直接可见，无需 DevMode）
- 每个扩展的配置编辑入口
- 全局扩展启用/禁用（远期）

### 2.2 架构：谁提供什么

```
宿主 (Client)
├── SettingsWindow         ← 通用连接设置（不变）
├── ServerTab              ← 纯容器，收集 IMainTabProvider
│   ├── [Chat Tab]         ← 来自 Chat 扩展
│   ├── [Trade Tab]        ← 来自 Trade 扩展
│   └── [扩展管理 Tab]     ← 新增，由宿主内置提供
└── ExtensionManagerTab    ← 实现 IMainTabProvider，宿主自身注册
```

**设计决策：扩展管理 Tab 由宿主内置实现，不走插件。**

理由：
- 扩展管理是"元能力"——它管理插件的生命周期，不应该由一个插件来管理其他插件（那会打破平权，制造超级公民）
- 宿主提供通用服务（设计哲学 1.3），扩展管理正属此类
- 保证没有任何插件时，扩展管理面板仍可达
- ExtensionManagerTab 通过 `IMainTabProvider` 接入——和所有插件 Tab 使用同一个挂载点，平等渲染，不创建宿主特权 UI 路径

### 2.4 实现路径

#### Step 2.4.1：`DiscoveredPhinixExtensions` 数据结构增强

```csharp
// FrameworkTypes.cs 或新文件 ExtensionDiscoveryResult.cs
public sealed class ExtensionDiscoveryResult
{
    public string ExtensionId { get; set; }
    public string DisplayName { get; set; }       // 新增：人类可读名称
    public string Version { get; set; }           // 新增：从 Assembly.GetName().Version
    public string AssemblyName { get; set; }       // 新增：来源程序集
    public ExtensionModuleState State { get; set; } // 新增
    public string StateDetail { get; set; }        // 新增：失败原因
    public string SourcePackageId { get; set; }     // 新增：来源 Mod 的 packageId（内置扩展同 Phinix 的 packageId）
    public List<string> Capabilities { get; set; }
    public List<string> RegisteredApis { get; set; }
    public List<string> ConsumedApis { get; set; }  // 通过 TryResolveApi 消费的 API
}

public enum ExtensionModuleState
{
    Unknown,
    Discovered,
    Registered,     // Register() 成功
    Active,         // Activate() 成功
    Failed,         // 加载/注册/激活失败
    Shutdown        // Shutdown() 已调用
}
```

#### Step 2.4.2：`PhinixExtensionRegistry` 增强

- `DiscoverExtensions()` 返回 `DiscoveredPhinixExtensions` 时填充 `ExtensionDiscoveryResult` 列表
- 记录每个扩展的来源程序集、版本号、来源 Mod 的 packageId
- `Register()` 和 `Activate()` 的结果映射到 `ExtensionModuleState`
- **错误隔离：** 每个扩展的 `Register()`/`Activate()`/`Shutdown()` 在独立的 try-catch 中执行。单个扩展的失败记录到日志并映射为 `Failed` 状态，不得中断其他扩展的加载流程（设计哲学 3.5）。

#### Step 2.4.3：`PhinixFrameworkClient` 暴露发现结果

```csharp
// 新增公共属性
public IReadOnlyList<ExtensionDiscoveryResult> ExtensionResults { get; }
```

在构造函数中从 `discoveredExtensions` 构建此列表，随 `Client.Instance.FrameworkClient.ExtensionResults` 可访问。

#### Step 2.4.4：`ExtensionManagerTab` 实现

- 新建 `Client/Source/GUI/Windows and panels/ExtensionManagerPanel.cs`
- 实现 `IMainTabProvider`（`TabLabel = "扩展"`, `TabOrder = 999`——排在最后）
- 在 `Client` 构造函数中注册：`builder.RegisterApi<IMainTabProvider>(new ExtensionManagerTab())`
- 渲染逻辑：
  - 上半部分：扩展列表（可滚动），每行显示状态图标、名称、版本、来源、操作按钮
  - 下半部分：日志面板，显示 `Diagnostics` 和 `Warnings`
  - "设置"按钮 → 弹出该扩展的配置窗口（见第三节）
  - "刷新"按钮 → 重新扫描扩展（热重载的铺垫）

#### Step 2.4.5：Warnings 始终可见

- `DiscoveredPhinixExtensions.Warnings` 中 **所有 Warnings（包括加载失败）** 在扩展管理面板中始终以 `WARNING` 级别显示红色文本
- 不再依赖 DevMode 开关
- 启动时如果有 Error/Warning 级别的诊断，在主界面显示一次性通知气泡："N 个扩展存在问题，点击查看"

## 3. 扩展设置管理设计

### 3.1 目标

让每个扩展通过统一机制暴露自己的配置 UI，而非在 `SettingsWindow` 中硬编码 Chat/Trade 专属 UI。

### 3.2 接口设计

```csharp
// 可选接口，扩展实现即可在扩展管理面板中出现 [设置] 按钮
public interface IExtensionSettingsProvider
{
    string SettingsCategory { get; }  // 设置分类名
    void DrawSettings(Rect inRect);   // 绘制设置 UI
}
```

扩展在 `Register()` 中注册：

```csharp
public void Register(IExtensionBuilder builder)
{
    // ... 其他注册 ...
    builder.RegisterApi<IExtensionSettingsProvider>(new MySettingsProvider());
}
```

`ExtensionManagerTab` 收集所有 `IExtensionSettingsProvider`，为每个提供"设置"按钮，点击后弹出 `ExtensionSettingsWindow`。

### 3.3 与现有 SettingsWindow 的关系

**迁移路径：**

1. Chat 扩展实现 `IExtensionSettingsProvider`，将当前在 `Client.DoSettingsWindowContents()` 中的 Chat 专属设置移过去
2. Trade 同
3. `Client.DoSettingsWindowContents()` 精简为仅保留通用设置：地址、端口、显示名、音效
4. `SettingsWindow` 底部加一个"扩展设置"按钮，跳转到扩展管理面板

### 3.4 设置持久化

扩展设置通过 `IClientSettingsContext.Get<T>()` / `Set<T>()` 读写，已自动持久化到 RimWorld ModSettings（`ExposeData` 中的 `ExtensionSettings` 字典）。无需额外改动。

## 4. 版本兼容性（远期考量）

> **当前状态：** Phinix 首个正式版刚发布，无外部开发者，尚未发生框架 API breaking change。以下内容为远期设计方向，当前不实现。

### 4.1 设计方向

当框架 API 发生第一个 breaking change（如移除或修改 `IExtensionBuilder` / `IExtensionHostContext` / `IClientSettingsContext` 等方法签名）时，引入**新接口** `IFrameworkVersionConstraint`（放在 `ClientExtensionAbstractions` 中）：

```csharp
// 新接口，不修改 IPhinixExtensionModule
public interface IFrameworkVersionConstraint
{
    // 声明此扩展支持的最小框架 API 版本
    int MinFrameworkApiVersion { get; }
}
```

框架在 `DiscoverExtensions()` 中检查：

```csharp
const int CurrentFrameworkApiVersion = 1;

if (module is IFrameworkVersionConstraint versioned && 
    versioned.MinFrameworkApiVersion > CurrentFrameworkApiVersion)
{
    discovered.Warnings.Add(
        $"扩展 '{module.ExtensionId}' 需要框架 API v{versioned.MinFrameworkApiVersion}，" +
        $"但当前框架仅为 v{CurrentFrameworkApiVersion}。请更新 Phinix。");
    continue;
}
```

### 4.2 设计原则

- **新接口，不改旧接口。** `IPhinixExtensionModule` 签名不变，符合 Section 7 稳定性承诺。
- **不实现的扩展完全不受影响。** 不实现 `IFrameworkVersionConstraint` = 不参与版本检查。
- **`CurrentFrameworkApiVersion` 仅在发生 breaking change 时递增**。新增服务、新增 Builder 方法不递增。

### 4.3 当前做法

在实现此机制前，兼容性通过以下方式保障：
- `About.xml` 中的 `modDependencies` 声明 Phinix 依赖
- 发布说明 / changelog 中标注 breaking change 迁移要点

## 5. Starter 项目模板

### 5.1 独立仓库结构

```
Phinix-Plugin-Template/
├── README.md                        ← 文档（即本文档的精简版）
├── About/
│   └── About.xml                    ← Mod 元数据
├── Assemblies/
│   └── .gitkeep
├── Source/
│   ├── MyExtension.cs               ← 最小可运行扩展示例
│   ├── MyExtension.Client.csproj     ← net472 引用 Utils.dll
│   └── Properties/
│       └── AssemblyInfo.cs
├── Languages/
│   ├── English/
│   │   └── Keyed/
│   │       └── Core.xml
│   └── ChineseSimplified (简体中文)/
│       └── Keyed/
│           └── Core.xml
└── .gitignore
```

### 5.2 关键文件内容

**About/About.xml：**
```xml
<ModMetaData>
    <name>Phinix Plugin Template</name>
    <packageId>yourname.phinix-plugin</packageId>
    <author>YourName</author>
    <supportedVersions><li>1.6</li></supportedVersions>
    <modDependencies>
        <li><packageId>hunyuan2333.phinixrework</packageId></li>
        <li><packageId>brrainz.harmony</packageId></li>
    </modDependencies>
</ModMetaData>
```

**MyExtension.cs 示例：**
```csharp
using System.Collections.Generic;
using Utils;
using Utils.Framework;
using Verse;
using UnityEngine;

namespace MyPlugin
{
    [PhinixExtension("my.first_plugin")]
    public class MyExtension : IPhinixExtensionModule, IActivatablePhinixExtensionModule,
                                 ICapabilityProvider, IClientMessageHandler, IMessageRenderer
    {
        public string ExtensionId => "my.first_plugin";
        public int Priority => 100;

        public void Register(IExtensionBuilder builder)
        {
            builder.AddCapabilityProvider(this);
            builder.AddClientMessageHandler(this);
            builder.AddMessageRenderer(this);

            // 读取设置（自动持久化）
            string greeting = builder.HostContext
                .GetRequiredService<IClientSettingsContext>()
                .Get("my.greeting", "Hello!");
            Log.Message($"[MyPlugin] Greeting: {greeting}");
        }

        public void Activate(ExtensionHostContext hostContext) { }
        public void Shutdown(ExtensionHostContext hostContext) { }

        public IEnumerable<string> GetCapabilities()
        {
            yield return ExtensionId;
        }

        // ... 消息处理逻辑 ...
    }
}
```

### 5.3 `.csproj` 要点

- `TargetFramework`: `net472`（Client 侧）
- 引用 `Utils.dll`：通过相对路径指向 RimWorld Mods 目录中的 Phinix 安装
- 引用 `ClientExtensionAbstractions.dll`：同上
- 引用 RimWorld 基础 DLL：`Assembly-CSharp.dll`、`UnityEngine.dll` 等
- 构建后拷贝到 RimWorld Mods 目录

## 6. 优先级与执行建议

| 优先级 | 阶段 | 内容 | 预估工时 | 依赖 |
|--------|------|------|---------|------|
| P0 | 1 | 扩展管理面板（2.4.1-2.4.5） | 8-12h | 无 |
| P0 | 1 | 加载失败可见提示（2.4.5） | 1h（含在面板内） | P0-1 |
| P1 | 2 | IExtensionSettingsProvider 接口 + 统一设置入口 | 4-6h | P0-1 |
| P1 | 2 | Chat/Trade 设置从 SettingsWindow 迁出 | 4-6h | P1-1 |
| P2 | 3 | Starter 项目模板（独立仓库） | 3-4h | 无 |
| P3 | 远期 | 版本兼容性检查（IFrameworkVersionConstraint） | 等第一个 breaking change | - |
| P3 | 4 | 扩展热重载 | 推迟到 2.0 | - |
| - | 全阶段 | 开发者文档 | 与实现同步 | - |

**阶段 1 是核心**——扩展管理面板解决了最紧迫的问题：
1. 开发者能看见自己的扩展是否被识别、是否有错误
2. 玩家能看见哪些扩展在运行
3. 不再依赖 DevMode 发现加载问题

阶段 2 完成"扩展设置不与宿主耦合"的最后一步。

## 7. 关于 hook 点稳定性的说明

以下接口自 Phase 5 解耦完成后，在可预见的未来**不应变更**（除非协议本身重大升级）：

| 接口 | 稳定性 | 说明 |
|------|--------|------|
| `IPhinixExtensionModule` | **稳定** | `Register(builder)` 是唯一的接入点 |
| `IActivatablePhinixExtensionModule` | **稳定** | `Activate/Shutdown` 生命周期语义已确定 |
| `IExtensionBuilder` | **稳定** | 只会新增方法（向后兼容），不会改签名 |
| `IClientSettingsContext` | **稳定** | Key-Value 模式已证明足够通用 |
| `IExtensionConfigProvider` | **稳定** | Server 侧配置已通过 SectionName 机制脱耦 |
| `IMainTabProvider` / `IServerSidebarProvider` / `IBadgeProvider` | **稳定** | UI 挂载点已足够通用 |

新增框架能力通过**新增接口**而非修改现有接口进行——插件不实现新接口则不受影响。
