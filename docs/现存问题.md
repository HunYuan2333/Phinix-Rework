## 1. TradeWindow Accept/Cancel 勾选不生效（已处理，待实机回归）

**现象**：一方勾选 accept 后，另一方可以看到对方的勾选状态（快照正常推送），但未勾选方无法勾选确认；已勾选方也无法取消勾选；取消交易按钮无效。多次尝试修复均未解决，根因尚未确定。

**已排除的假设**：

- ~~`NetClient.Send` 缺少 lock 导致 LiteNetLib 并发写坏~~ — 已加 lock，现象依旧
- ~~交易完成/取消后 `Close()` 在非主线程执行~~ — 已修复为 `shouldClose` 标志在主线程关窗，现象依旧

**仍在排查的方向**：

### 1a. 勾选框每帧重置

`TradeWindow.DoWindowContents` 每帧从 `trade.Accepted` 重新初始化 `ourOfferAccepted`（TradeWindow.cs:201）：

```csharp
bool ourOfferAccepted = trade.Accepted;
```

用户点击勾选框 → `ourOfferAccepted` 翻转 → 发包 `UpdateTradeStatus`。下一帧 `trade.Accepted` 尚未更新（服务器还没返回快照），`ourOfferAccepted` 可能被重置回旧值，勾选框在 UI 上立刻恢复未勾选状态。用户可能以为没点上，再次点击 → 又发包 → 又重置 → 循环。

但此假设不能解释为什么服务器对 `UpdateTradeStatus` 请求**完全无响应**（服务端日志无任何 trade 相关记录），而 `UpdateTradeItems`（物品变更）的请求/响应正常。

### 1b. new Thread 发包行为异常

`UpdateTradeStatus` 和 `CancelTrade` 通过 `new Thread(() => ...).Start()` 发包（TradeWindow.cs:213,303），而 `UpdateTradeItems`（物品变更）直接在主线程调用（TradeWindow.cs:268,292）。

实测日志中 `UpdateTradeItems` → `HandleOfferUpdateResponse` 正常，但多次 `UpdateTradeStatus` → `HandleStatusUpdateResponse` 从未出现。**服务器端日志同样没有收到任何 StatusUpdate 请求**，与 `UpdateTradeItems` 形成鲜明对比。`new Thread` 可能是原因之一，但已尝试加 lock 无效，说明可能还有其他因素（如 `SendFrameworkPacket` 在非主线程被静默丢弃、`createContext()` 在子线程返回了异常状态等）。

### 1c. 其他待排查方向

- 服务端 `BuiltInTradeServerExtension` 是否正确处理 `StatusUpdateRequestType`？需要在服务端 trade handler 加日志确认
- `FrameworkTradeStatusUpdateRequest` 序列化/反序列化是否正确？`Accepted` 为 `bool?` 可能序列化行为与预期不同
- 是否与特定 trade 状态有关（如必须先 UpdateItems 才能 UpdateStatus）？
- 客户端和服务端是否使用了不同版本的 TradeExtension DLL？

**建议下一步**：在 `PhinixFrameworkClient.sendPacket` 中加日志确认 StatusUpdate 包是否真的发到了 LiteNetLib 层；在服务端 `BuiltInTradeServerExtension.HandleIncomingCommand` 加日志确认是否收到了 StatusUpdate 请求。

## 2. Chat 界面 Enter 键关闭 Phinix 主框（已处理，待实机回归）

**现象**：在 Chat 输入框中按 Enter，Phinix ServerTab 整个关闭而不是发送消息。

**根因**：RimWorld 的 `MainButtonWindow` 或 `ServerTab` 劫持了 Enter 键事件，Chat 插件未能拦截。

**修复方向**：Chat 插件的输入框需要在 `DoWindowContents` 中通过 `Event.current` 检查 `KeyCode.Return` 并调用 `Event.current.Use()` 阻止冒泡。

## 3. 服务器状态不会自动同步（已加刷新兜底，待实机回归）

**现象**：登录后需要手动断开再重连才能看到正确的在线用户列表和聊天历史。

**根因**：可能是 `IClientUserEventStream` 的 `UsersChanged` 事件在初始连接后未触发，或者 `UserList.refreshUserLists()` 未在 Activate 时调用。

**修复方向**：检查 `BuiltInChatClientExtension.Activate()` 和 `ChatSidebarProvider` 初始化流程，确保连接建立后立即拉取用户列表。

## 4. 客户端/服务端统一 Extensions 插件目录（已处理，待构建验证）

**现象**：Chat 和 Trade 插件 DLL 目前输出到 `Client/Common/Assemblies/` 和 `Server/` 目录，与框架基础 DLL 混在一起。Phase 5 的核心原则是插件平等——Chat 和 Trade 应与第三方 submod 走完全一样的路径。客户端和服务端都应该有独立的 `Extensions/` 文件夹。

**当前状态**：

| 端 | 插件 DLL 落位 | 问题 |
|----|-------------|------|
| 客户端 | `Client/Common/Assemblies/`（编号 08-11） | 混入框架基础 DLL；文件名排序决定了加载顺序，耦合脆弱 |
| 服务端 | `Server/bin/Debug/net10.0/` 编译产物目录 | TradeExtension.Server.dll / ChatExtension.Server.dll 与框架 DLL 平级，没有独立插件目录 |

**修复方向**：

- **客户端**：新增 `Client/Common/Extensions/` 目录，Chat/Trade 插件 DLL 和第三方 submod DLL 都放这里。`ExtensionAssemblyLoader` 已有探针目录机制（`GetExtensionProbeDirectories()`），加一个路径即可。插件 DLL 不再出现在 `Assemblies/` 里，RimWorld 的 `ModAssemblyHandler` 不会碰它们，彻底消除加载顺序问题。
- **服务端**：新增 `Server/Extensions/` 目录，TradeExtension.Server.dll / ChatExtension.Server.dll 编译后落位到此处。服务端 `ExtensionAssemblyLoader` 同样探针此目录自动发现。服务端插件的加载与激活应复用 `PhinixExtensionRegistry` 的反射扫描机制（与客户端一致）。
- **两者统一**：`Extensions/` 目录结构、发现机制、注册激活路径，客户端和服务端应完全一致。Chat 和 Trade 的 Server/Client 插件分别落位各自端的 Extensions 目录，第三方 submod 以同样的方式增加自己的 DLL 即可。
