# Phinix 设计哲学

> 面向代码维护者和 AI Agent 的架构指南。新增功能、修改行为、审查代码时，以此为基准。
> 2026-05-31，基于 Phase 5 架构迁移落点与代码库审计整理。

---

## 1. 核心原则

### 1.1 插件平权

Chat 和 Trade 与第三方 submod 地位相同。不存在"官方超级公民"：

- 同一套发现路径（反射扫描 `IPhinixExtensionModule`）
- 同一套注册路径（`Register(builder)` → `RegisterApi<T>()` / `AddClientMessageHandler()`）
- 同一套激活路径（`Activate(hostContext)` → `Shutdown(hostContext)`）

**反模式**：宿主为某个插件保留专用接口解析、专用启动分支、专用 UI 入口。

### 1.2 host 不依赖插件

宿主只引用 `ClientExtensionAbstractions`（通用契约层），不引用任何插件的 Contracts 工程：

- `Client.csproj` 的 ProjectReference 中**禁止出现** `ChatExtension`、`TradeExtension` 或任何插件工程
- 宿主通过 `ResolveExtensionApis<T>()` 动态收集插件能力，不做类型强绑定
- 插件可以依赖 host，反过来不行

**反模式**：`Client.csproj` 显式引用插件工程；宿主代码中出现 `IClientChatService`、`IClientTradeService` 等具体业务接口的类型引用。

### 1.3 host 只做通用服务

宿主提供的基础服务与业务无关：

```
网络层 (NetClient)
扩展发现 (PhinixExtensionRegistry)
通用服务 (IClientSessionContext, IClientSettingsContext, IClientUserDirectory,
           IClientUserEventStream, IClientMainThreadDispatcher, IClientWindowService)
ServerTab（纯壳，收集 IMainTabProvider / IServerSidebarProvider 动态渲染）
基础 UI (SettingsWindow, CredentialsWindow)
```

业务逻辑（聊天、交易、红包……）全部在插件里。宿主不关心当前有哪些业务。

## 2. 软件工程原则

### 2.1 松耦合

模块之间通过接口契约交互，不直接依赖具体实现：

- 插件与宿主通过 `ClientExtensionAbstractions` 中定义的通用接口耦合，不通过具体类型
- 插件间交互由插件自行定义接口并直接调用，框架不充当中介也不阻止
- 新增插件不影响宿主编译，不影响已有插件运行

**判断标准**：能否在不修改 host 代码的前提下，新增一个完整的业务插件（含 Tab、侧栏、角标）？

### 2.2 层次化

系统分为清晰的层次，上层依赖下层，禁止反向：

```
┌─────────────────────────────────────────┐
│  Plugins (Chat, Trade, Third-party)      │  ← 业务层
├─────────────────────────────────────────┤
│  ClientExtensionAbstractions             │  ← 共享契约层
├─────────────────────────────────────────┤
│  Host (Client)                           │  ← 宿主层
│  ├─ 网络 / 认证 / 用户管理               │
│  ├─ 扩展发现 / 生命周期                   │
│  └─ 通用 UI 壳                           │
├─────────────────────────────────────────┤
│  Common (Utils, Connections, etc.)       │  ← 基础设施层
└─────────────────────────────────────────┘
```

- 上层可以调用下层，下层**绝不**反向依赖上层
- 同一层内的模块尽量保持独立，减少横向耦合
- 每层只暴露该层职责范围内的接口，不泄露内部实现细节

**反模式**：`Common` 编译 `../../Server/*.cs` 源码；宿主代码直接访问插件内部类型。

### 2.3 减少硬编码

凡是可能随业务变化的，通过合约抽象，不写死在宿主：

- Tab 内容 → `IMainTabProvider`，不硬编码"聊天 Tab / 交易 Tab"
- 侧栏内容 → `IServerSidebarProvider`，不硬编码"用户列表面板"
- 角标 → `IBadgeProvider`，不硬编码"未读消息数"
- 消息处理 → `module → Kind → MessageType → handler` 动态路由，不针对特定消息类型硬编码分发
- 事件通知 → `IClientUserEventStream`，不针对特定插件提供专用事件桥接

**判断标准**：新增一个业务能力（如邮件、公告、任务），宿主代码改动是否为零？

## 3. 关键设计决策

### 3.1 动态 Tab 机制

通信层和 UI 层使用同一套动态分发模式：

| | 通信层 | UI 层 |
|---|---|---|
| 接口定义 | `FrameworkTypes.cs` | `IMainTabProvider.cs` |
| 插件注册 | `builder.AddClientMessageHandler(this)` | `builder.RegisterApi<IMainTabProvider>(this)` |
| host 收集 | `foreach handler in ClientMessageHandlers` | `foreach provider in MainTabProviders` |
| 路由依据 | `CanHandleIncomingMessage(message)` | `TabOrder` 排序，用户点击切换 |

- `ServerTab` 是纯容器，不包含任何业务 UI
- `ServerTabButtonWorker` 聚合所有 `IBadgeProvider`，只显示第一个有内容的角标
- 新增加 Tab/侧栏只需实现对应接口并在 `Register()` 中注册

### 3.2 通信管道三分类

框架通信分为三条主链，各司其职：

| 管道 | 职责 | 示例 |
|------|------|------|
| Message | 用户可见消息 | 聊天消息、系统通知 |
| Command | 后台控制指令 | 交易创建/更新、快照同步 |
| Item | 物品数据 | 交易物品编码/解码 |

新增消息类型时，先判断属于哪条管道，再选择对应的 handler 接口实现。

### 3.3 插件间交互

插件之间的协作由插件自行负责，框架不充当中介：

- Chat 需要发起交易 → Chat 直接引用 `TradeExtension` 的 Contracts，调用 `ITradeRequestApi.CreateTrade()`
- 两个插件需要共享数据 → 各自定义接口，通过 API registry 互相解析
- 框架的作用只是提供发现机制（API registry），不承担业务协调

### 3.4 通用事件流取代专用桥接

插件不通过宿主专为其定制的桥接接口获取宿主事件，而是消费通用事件流：

- `IClientUserEventStream`：断线、用户变更、改名、屏蔽状态变更
- `IClientMainThreadDispatcher`：从网络线程回到主线程
- `IClientWindowService`：打开宿主窗口（如设置窗口）
- `IClientSettingsContext`：读写设置
- `IClientSessionContext`：获取当前会话状态

**反模式**：宿主定义 `IChatUiEventSink`、`ITradeUiHostContext` 等插件专用接口，每次新增插件就新增一套。

### 3.5 错误隔离与重试机制

网络处理、消息解析、扩展执行中的单个失败**不得**中断整体管线或导致服务崩溃。框架对各类错误采取分层隔离策略：

**管线内隔离：**

- 单个拦截器/处理器抛出的异常由 PipelineRunner 捕获并记录，管线继续处理下一个候选者
- 单个消息的解析失败（如 Protobuf `ParseFrom` 异常）仅跳过该消息，不影响同一帧内的其他消息
- 扩展的 `Activate()` / `Shutdown()` 失败不影响其他扩展的生命周期

**网络层韧性：**

- 关键握手包（Hello / Auth / Login / ExtendSession）发送失败必须主动断开对应连接，防止客户端无限等待
- 非关键消息发送失败记录错误日志并通知调用方（通过返回值或回调），不静默丢弃
- 连接断开时立即清理关联的 session、用户状态、事件订阅，不依赖超时定时器兜底

**重试策略：**

- 客户端连接失败应提供明确的错误信息（区分"网络不可达""认证被拒""协议不匹配"），而非笼统的"连接失败"
- 客户端未连接时发送的消息应触发可观测的告警（日志级别 Error 或回调通知），不静默丢弃
- 服务端向特定连接发送失败时区分"连接已断开"和"发送超时"——前者触发清理流程，后者可考虑重试

**反模式**：catch-all 静默吞掉异常；后台线程异常不通知主线程；发送失败仅写一行 Debug 日志而调用方无感知。

### 3.6 反压与资源边界

所有无界队列必须有容量上限，防止生产者快于消费者时内存无限增长：

- 消息显示队列（`displayMessages`）：上限 1000，超出时移除最旧的条目并记录 Warning
- 主线程调度队列（`pendingActions`）：上限 500，超出时丢弃并记录 Error
- 聊天历史同步：分批发送（每批 50-100 条），不在单次 poll 周期内阻塞发送全部历史

持有 `Timer`、`NetManager`、`Thread`、`FileStream` 等 `IDisposable` 资源的类必须实现 `IDisposable` 并在 `Shutdown` / `Dispose` 中释放。事件订阅（`+=`）必须在对应的 `Shutdown` 或 `Dispose` 中配对取消（`-=`），不依赖进程退出作为终极清理手段。

### 3.7 插件不得绕过通信管线直连底层传输

插件与服务器通信时，**必须通过框架定义的 handler 管线**（`IClientMessageHandler`、`IClientCommandHandler`），**不得**直接调用 `IFrameworkClientTransport.SendFrameworkPacket()` 或 `ILegacyModuleTransport.Send()` 绕过管线。

**为什么：**

- handler 管线是框架实现"插件平权"（§1.1）的机制 —— Priority 排序、拦截、替换、回退都依赖 handler 依次执行
- 插件直接发 FrameworkPacket 绕过管线 = 该协议包**跳过所有其他插件的 handler**。如果另一个插件（如 LegacyAdapter）通过 Priority 排序尝试拦截流量做协议翻译，它将完全无效
- 一个生态位良好的插件（如消息审计、内容过滤、协议适配）不应该因为其他插件"走了捷径"而失效

**正确做法：**

- 出站消息：通过 `IClientMessageHandler.HandleOutgoingText()` 返回 `FrameworkPacket`，让 `PhinixFrameworkClient.TryHandleOutgoingMessage()` 统一发送
- 出站命令（Trade等）：通过 `IClientCommandHandler` 管线，或通过 `IFrameworkClientLifecycle.CompatibilityMode` 判断当前模式后决定路由策略
- 入站消息：由框架从 `NetClient` 接收后分发给 handler 管线；插件不自行注册 `NetClient` handler 拦截入站（Legacy 协议适配除外，因为 Legacy 入站完全不在 Framework 管线上）

**唯一例外**：Legacy 协议适配器可以通过 `ILegacyModuleTransport.RegisterHandler()` 注册原始模块 handler，因为 Legacy 入站数据完全不经过 Framework 管线（老服务器发送的是 `"Chat"` / `"Trading"` 模块包，不是 `"PhinixFramework"` 包）。

**判断标准**：任意一个新插件能否通过 Priority 排序插入到通信管线中，拦截/修改/替换消息而不需要其他插件主动配合？

### 3.8 日志与可观测性

框架通过 `ILoggable` 接口（定义于 `Common/Utils/ILoggable.cs`）提供统一的日志事件机制。所有 host 组件和插件应通过此接口上报日志，而非绕过框架自行写入控制台或文件。

**日志级别约定：**

| 级别 | 含义 | 使用场景 |
|------|------|----------|
| `DEBUG` | 开发诊断信息 | 管线消息路由细节、扩展加载过程、临时调试 |
| `INFO` | 正常运行信息 | 连接建立/断开、扩展激活/关闭、配置加载 |
| `WARNING` | 可恢复的异常 | 队列溢出、消息解析失败但管线继续、重试成功 |
| `ERROR` | 需关注的故障 | 握手包发送失败、扩展激活失败、发送超时 |
| `FATAL` | 服务不可继续 | 关键资源耗尽、不可恢复的状态损坏 |

**日志内容规范：**

- 所有异常日志必须附带 `Exception` 对象（如 `LogEventArgs` 支持），不能只记录 `ex.Message`
- 网络消息日志**不得**记录明文 token、密码、密钥等敏感字段；协议包内容记录需脱敏
- 消息级日志（如管线观察器 `IServerMessageObserver`）应包含消息类型标识，便于按模块过滤

**生产环境最低可观测性要求：**

- 当前连接数（`connectedPeers.Count`）应在日志中周期性可见或可通过命令查询
- 消息吞吐量（入站/出站 QPS）应可通过 `IServerMessageObserver` 汇总后接入日志
- 错误率：至少区分"管线内业务错误"和"网络层传输错误"，分别统计

**客户端侧特别约定：**

- RimWorld 控制台会吞掉部分 stdout 输出，客户端日志不应仅依赖 `Console.Write`——应通过 `ILoggable` 事件桥接到宿主的标准日志路径
- Debug 级别日志默认不输出到 RimWorld 控制台，避免淹没用户

**反模式**：插件绕过 `ILoggable` 自行 `Console.WriteLine`；异常日志只写 `ex.Message` 丢弃堆栈；生产日志级别设为 DEBUG。

## 4. 边界规则

### 4.1 引用方向

```
Client → Common (shared contracts & abstractions) ← Server

禁止：Common → ../../Server/*.cs
禁止：Common → ../../Client/*.cs
禁止：Client → 具体插件工程 (ChatExtension, TradeExtension, ...)
```

### 4.2 Common 能放什么

- 协议（protobuf contracts, packet DTO）
- 抽象（extension module contracts, handler contracts, API registry abstractions）
- runtime-neutral 工具（序列化、文本处理、基础日志接口）
- 双端都需要且不绑定特定运行时的基础设施

### 4.3 Common 不能放什么

- 业务处理器（聊天、交易、或任何具体业务逻辑）
- 服务端状态管理器
- 客户端 UI 模型
- 任何 client-only 或 server-only 运行时实现
- host 对官方插件的特殊假设

### 4.4 判断标准

> 如果一段代码移到某一端（Client 或 Server），另一端**完全不受影响**，那它就不该在 Common 里。

### 4.5 Common 源文件共享模式（现状过渡）

当前部分端专属实现（`ClientAuthenticator.cs`、`NetClient.cs`、`NetServer.cs`、`ClientUserManager.cs` 等）物理上存放于 `Common/` 目录下，但通过 `.csproj` 中的 `Compile Remove` / `Compile Include` 链接机制确保**编译归属是单端的**——Common 的 `Connections.csproj` 排除 `NetClient.cs` 和 `NetServer.cs`，由 `Connections.Client.csproj` 和 `Connections.Server.csproj` 分别通过路径链接编译。

此模式是过渡态，物理位置 ≠ 编译归属。判断文件是否越界的标准是**编译归属**而非物理路径：

- 一个文件被 `Common/Authentication/Authentication.csproj` 编译 → 属于 Common
- 一个文件仅被 `Client/Common/Authentication.Client/Authentication.Client.csproj` 编译 → 属于 Client，不管它物理上放在哪里

**目标态**：端专属实现物理移动到各自端目录下，Common 目录中仅保留真正 runtime-neutral 的源文件。此调整应在程序集拆分（§5.2）完成后进行。

## 5. 目录与程序集约定

### 5.1 命名与排序

`Client/Common/Assemblies/` 中的 DLL 使用零填充数字前缀确保字符串序 = 加载序：

```
01-LiteNetLib.dll         ← 第三方库
02-Protobuf.dll
03-Utils.dll               ← IPhinixExtensionModule, framework 基础
04-Connections.dll
05-Authentication.dll
06-UserManagement.dll
07-ClientExtensionAbstractions.dll  ← IMainTabProvider, IServerSidebarProvider, 通用 host 服务
08-ChatExtension.dll       ← Chat 领域契约
09-TradeExtension.dll      ← Trade 领域契约
10-ChatExtension.Client.dll  ← Chat 插件（依赖 03, 07, 08）
11-TradeExtension.Client.dll ← Trade 插件（依赖 03, 07, 09）
```

新增 DLL 时按依赖关系分配编号。当且仅当字符串序保证所有依赖先于被依赖方加载，RimWorld 的 `ModAssemblyHandler` 才不会抛出 `ReflectionTypeLoadException`。

### 5.2 发布边界（目标态）

当前官方插件 DLL 与框架基础 DLL 仍混放在 `Assemblies/` 中。最终目标：

```
Client/
  Common/
    Extensions/          ← 插件 DLL 独立目录（RimWorld 不碰它）
      ChatExtension.Client.dll
      TradeExtension.Client.dll
      SomeSubMod.dll
    Assemblies/          ← 仅框架基础 DLL（01-07）

Server/
  Extensions/            ← 服务端插件 DLL 独立目录
    ChatExtension.Server.dll
    TradeExtension.Server.dll
```

这个调整的前提是 `ExtensionAssemblyLoader` 能从此目录加载、`PhinixExtensionRegistry` 能据此发现——机制已就绪，只是目录结构和构建链需对齐。

### 5.3 版本化与 API 兼容

**程序集版本统一：**

所有程序集版本号通过 `Directory.Build.props` 集中管理，不分散到各 `AssemblyInfo.cs` 手动维护。当前各项目版本不一致（如 `Server.csproj` 为 `0.1.0.0`，`Connections.Server` 为 `1.0.0.0`），需对齐为统一版本。

**Git 标签与发布：**

- 语义化版本（Semver）：`MAJOR.MINOR.PATCH`
  - `MAJOR`：破坏性 API 变更（如接口删除、方法签名改变）
  - `MINOR`：向后兼容的新增（新接口、新方法、新插件挂载点）
  - `PATCH`：纯修复，不改变任何公开 API
- 每次发布打 Git 标签 `v<semver>`，触发 Docker 镜像构建与发布

**Protobuf 消息兼容：**

- 字段编号（field number）是**永久的、不可更改的**。删除字段必须用 `reserved` 保留编号和名称，防止未来重用
- 新增字段只能追加，不得插入已有字段之间
- 枚举值不得删除或重编号——废弃值标记 `[Obsolete]` 或注释，但保留其数值
- 修改消息类型语义时必须创建新消息类型（如 `LoginRequestV2`），旧类型作为遗留兼容保留

**API 废弃生命周期：**

与 §6 Host/Core 增量更新规则配套，`[Obsolete]` 标记遵循以下节奏：

```
标记 [Obsolete] → 至少保留 1 个 MINOR 版本 → 确认无下游引用后移除
```

- 废弃接口必须在 XML 文档注释中指明替代方案（`<summary>Use IXxx instead.</summary>`）
- 移除前应搜索整个仓库及已知第三方 submod 确认无残留引用
- `MAJOR` 版本升级时可以集中清理已到期废弃项

## 6. 渐进式迁移原则

- **每阶段保持可编译、可运行、可验证**。不做"一次性大迁徙"。
- **先收口边界，再做运行时迁移，最后完整插件化**。
- **重构不改行为**。Phase 5 的每次改动（类型移动、引用调整）只改变代码归属，不改变运行时行为。如果行为变了，就是 bug。
- **目录收口先于程序集拆分**。物理目录可以先整理干净，程序集边界可在后续阶段逐步收紧。
- **新代码遵从新规则，旧代码逐步迁移**。新增功能必须满足当前阶段的边界要求；已有代码按优先级逐步迁出。
- **Host 与 Core 仅做增量更新**。对 Host 和 Core 的所有改动必须为增量式——不删除、不移除任何现有公开接口。如需破坏性变更，保留原接口并标记 `[Obsolete]`（或注释 `// outdated`），内部实现可重写但外部行为必须一致。此规则确保下游插件和第三方 submod 不因框架升级而编译失败或运行时断裂。

## 7. 提交与审查 check-list

新增或修改代码时，检查以下几点：

- [ ] host 工程是否新增了对插件的引用？
- [ ] Common 是否新增了 client-only 或 server-only 的代码？如确需新增，是否仅通过 `.csproj` 链接编译归属，而非直接编译进 Common 程序集？
- [ ] 是否硬编码了某个具体业务类型（如直接使用 `IClientChatService`）而非通过通用接口（如 `IMainTabProvider`）？
- [ ] 新增的扩展入口是否通过现有通用挂载点接入（Tab/侧栏/角标/消息处理），还是在宿主里开了新的专用口子？
- [ ] 插件间交互是否直接在插件间完成，而不是通过 host 中转？
- [ ] 字符串排序下的 DLL 加载顺序是否正确？
- [ ] 新增的网络处理/消息解析是否有 try-catch 隔离？单个消息失败是否会中断整个管线？
- [ ] 新增的 `IDisposable` 资源持有者是否实现了 `IDisposable`？事件订阅是否在 `Shutdown` 中配对取消？
- [ ] 新增的队列/缓冲区是否有容量上限？是否处理了生产者快于消费者的场景？
- [ ] 对 Host/Core 的改动是否为增量式？是否删除或移除了任何现有公开接口？如需破坏性变更，是否保留了原接口并标记 `[Obsolete]`？
- [ ] 日志调用是否通过 `ILoggable` 接口上报，而非绕过框架自行写控制台/文件？异常日志是否附带完整的 `Exception` 对象？
- [ ] 是否新增了公开 API？如新增，版本号是否需要升级为 `MINOR`？如修改/移除，是否已标记 `[Obsolete]` 并在文档中指明替代方案？
- [ ] Protobuf 字段变更是否兼容——未重用已删除字段编号、未修改枚举值、破坏性变更是否创建了新消息类型？

## 8. 发布前性能与稳定性审查

每个里程碑发布前，对照以下类别逐项审查。审查通过标准：**所有 CRITICAL 项清零，HIGH 项有明确处置计划**。

### 8.1 内存泄漏

**审查要点：**

- 所有 `IDisposable` 资源持有类是否在 `Shutdown` / `Dispose` 中释放？检查对象：`Timer`、`NetManager`、`Thread`、`FileStream`、`StreamReader`/`StreamWriter`
- 事件订阅是否有配对的取消订阅（`+=` ↔ `-=`）？依赖进程退出作为唯一清理手段视为未通过
- 字典/列表等集合在元素移除时键是否匹配？（如 session 字典用 connectionId 做键但 Remove 用了 sessionId）

**检查方法：** 启动服务器 → 连接 5 个客户端 → 断开 → 重复 3 轮 → 检查内存基线是否持续上升。客户端同理：进入服务器 → 切换 Tab → 退出 → 重复。

**反模式**：依赖超时定时器兜底清理本应在事件触发时立即清理的资源。

### 8.2 错误处理

**审查要点：**

- 网络层：`connectedPeers` / `probePeers` 的读写是否有锁保护？后台线程触发的回调是否封送至主线程？
- 管线层：单个拦截器/处理器异常是否被 PipelineRunner 捕获？异常后管线是否继续处理下一个候选者？
- 解析层：Protobuf `ParseFrom`、`Unpack` 等反序列化操作是否有 try-catch？恶意或损坏的消息是否仅跳过而不中断管线？
- 关键握手包：Hello / Auth / Login / SessionExtend 的响应发送失败是否导致了连接断开和资源清理？
- 客户端断线：未连接时发送的消息是否触发了 Error 级别日志或回调通知，而非静默丢弃？
- 异常粒度：catch-all（裸 `catch` 或无异常类型过滤）是否替换为捕获特定异常类型？`catch` 作为流控是否已消除？

**检查方法：** 使用损坏的 protobuf 数据包发送到服务端，确认管线继续处理后续正常消息。强制断开客户端网络，确认服务端在超时后清理 session 和用户状态。

### 8.3 UI 渲染性能

**审查要点：**

- Per-frame 分配：`DoWindowContents` / `Draw` / `DoButton` 路径上是否有 `new` 对象？（Regex、TextWidget、GUIContent、List、HeightContainer 等）
- Regex：所有用于富文本标签剥离的 Regex 是否为 `static readonly` 预编译实例？（`RegexOptions.Compiled`）
- 布局缓存：`CalcHeight` / `CalcWidth` 是否在数据变更时计算一次并缓存，而非在 Draw 路径上每帧重新计算？
- LINQ 分配：Draw 路径上是否有 `.Where()` `.Sum()` `.Select()` `.ToList()` `.Count()` 等分配枚举器的 LINQ 调用？
- 属性 getter：属性 getter 是否在每帧被调用（如 `BadgeText`）且每次都执行计算或分配新对象？是否改为推送式缓存更新？
- 滚动列表：大型列表（聊天消息、交易物品）是否每帧遍历全部条目计算布局？是否引入脏标记跳过无变更时的重计算？

**检查方法：** RimWorld 开发者模式 → 打开 Performance Profiler → 进入服务器 Tab → 发送 100 条聊天消息 → 上下滚动 → 检查 GC.Alloc 和帧时间。每帧 GC 分配应接近零。

### 8.4 线程安全

**审查要点：**

- 共享集合（Dictionary、List）的读写是否有 `lock` 保护或使用 `ConcurrentDictionary` / `ConcurrentQueue`？
- 网络回调（`OnPeerConnected`、`OnNetworkReceive` 等）在 poll 线程上触发时，操作 UI 或共享状态的代码是否封送至主线程？
- `Connected` 检查与 `Send` 之间是否存在 TOCTOU 竞态？是否使用 `TrySend` 代替 `Send`？

**检查方法：** 高并发连接测试（10 客户端同时连接+发送消息）→ 检查无 `InvalidOperationException`（集合已修改）、无崩溃。
